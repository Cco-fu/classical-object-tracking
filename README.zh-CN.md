# OTB100 传统目标追踪实验项目

中文 | **[English](./README.md)**

一个基于 **传统（非深度学习）单目标追踪方法** 的从零实现小项目，在
[OTB100](http://cvlab.hanyang.ac.kr/tracker_benchmark/) 基准数据集的若干序列上进行了测试。这个项目最初是作为学习练习搭建的，目的是理解在深度学习追踪器成为主流之前，"传统"计算机视觉方法（背景差分、帧间差分、颜色直方图、边缘模板匹配）能做到什么程度、又会在哪里失效。

项目实现了四条追踪流水线，均遵循相同的 `Tracker + Scorer` 结构，方便公平对比：

| 追踪器 | 文件 | 核心思路 |
|---|---|---|
| 帧间差分 | `frame_diff_tracker.py` | 相邻两帧的绝对差分 |
| 背景差分 | `background_diff_tracker.py` | 与一个缓慢更新的背景模型做绝对差分 |
| 颜色直方图 | `color_thresh_tracker.py` | 对目标 HSV 颜色直方图做反向投影（类 CamShift 思路） |
| 边缘模板 | `edge_detect_tracker.py` | Canny 边缘图 + 在局部搜索窗口内做模板匹配 |

这些方法都不是最先进的追踪算法——它们被有意设计得简单、可解释、也容易失效，这正好让人能直观"感受"到促使更现代追踪方法出现的那些失败模式：遮挡、漂移、模板污染、光照变化等等。

## ⚠️ 项目状态 / 诚实说明

这是探索性质、偏课程作业风格的代码，不是打磨好的库。在深入之前有几点需要了解：

- 各个 Tracker 类之间结构高度相似（生成掩码 → 候选打分 → 模板更新），但**目前尚未重构成统一的基类**。欢迎提交减少重复代码的 PR。
- 项目实现了模板自适应更新，但从实验结果看，**更新不总是有益的**——在不少序列上，一个**冻结不变**的初始模板反而比自适应更新的模板效果更好，因为一次错误匹配就可能"污染"模板，之后追踪器很难自我修正。这个权衡在下面 [关于模板更新的说明](#关于模板更新的说明) 部分有更详细的讨论。
- 这些追踪器都不太能很好地处理尺度变化、旋转，或长时间的完全遮挡。它们是为了学习/对比设计的，不是面向生产环境的。

## 项目结构

```
.
├── background_diff_tracker.py   # 背景差分追踪器 + 演示脚本
├── color_thresh_tracker.py      # HSV 颜色直方图反向投影追踪器 + 演示脚本
├── edge_detect_tracker.py       # Canny 边缘模板追踪器 + 演示脚本
├── frame_diff_tracker.py        # 帧间差分追踪器 + 演示脚本
└── utils/
    ├── loader.py                 # 多线程 OTB100 序列/帧加载器
    ├── selectors.py               # "Scorer" 类：把候选框打分并选出最终预测
    └── statistic.py               # IoU / CLE / 成功率评测
```

### 各部分是如何配合工作的

1. **`OTB100Loader`**（`utils/loader.py`）在后台线程里持续从 OTB100 序列文件夹读取 `(frame, ground_truth_box)` 数据对，避免帧解码阻塞主追踪循环。
2. 每个 **Tracker** 类（比如 `EdgeDetectTracker`）用自己的方法（边缘、颜色反向投影、帧差等）把当前帧转换成一张二值"掩码"（mask），再把这张掩码交给一个 **Scorer**。
3. 每个 **Scorer** 类（在 `utils/selectors.py` 中）负责把掩码转换成一个最终预测框：先找出候选区域，再结合*外观相似度*（与存储的模板比较）和*运动连续性*（与上一帧预测位置的距离）对候选框打分排序。
4. **`Statistic`**（`utils/statistic.py`）逐帧累积 IoU 和中心位置误差（CLE），并在追踪结束后打印汇总统计（均值/中位数/标准差、最佳/最差帧、给定 IoU 阈值下的成功率）。

项目中边界框统一采用 `(x, y, w, h)` 格式（左上角坐标 + 宽高），与 OTB100 的 ground-truth 格式保持一致。

```bash
git clone https://github.com/Cco-fu/classical-object-tracking.git
cd classical-object-tracking
pip install -r requirements.txt
```

## 获取数据集

本项目需要本地已下载好的 [OTB100 数据集](http://cvlab.hanyang.ac.kr/tracker_benchmark/benchmark_v10.html)，每个序列的目录结构应为：

```
OTB100/
└── Walking/
    ├── img/
    │   ├── 0001.jpg
    │   ├── 0002.jpg
    │   └── ...
    └── groundtruth_rect.txt
```

脚本默认在相对于项目目录的 `../OTB100` 路径下寻找数据集——请根据你本地数据集的实际位置和想测试的序列，调整每个追踪器脚本底部 `OTB100Loader(...)` 的路径参数（以及 `DATA_NAME` 常量）。

## 运行追踪器

每个追踪器文件本身也是一个可直接运行的演示脚本。例如：

```bash
python edge_detect_tracker.py
```

运行后会：
- 打开两个 OpenCV 窗口，分别显示追踪框与 ground truth 的对比，以及中间过程的掩码图；
- 在序列结束后打印 FPS 及汇总统计信息（平均 IoU、平均 CLE、成功率）。

在视频窗口中按 `Esc` 键可提前终止。

关键参数（核大小、二值化阈值、模板更新系数、搜索半径等）都在每个脚本底部的 `parameters` 字典里暴露出来——具体每个参数的含义可参考对应 `Tracker` 类的文档字符串。

### 调试模式

每个追踪器都配有一个 `Debug*` 子类（例如 `DebugEdgeDetectTracker`），会把中间结果图像（边缘图、掩码、轮廓、逐帧结果图）写入磁盘，保存在 `./output/<序列名>/frame<帧号>/` 下，便于直观排查追踪器在某个具体序列上失败的原因。可以通过每个脚本顶部的 `DEBUG` 开关来切换。

## 关于模板更新的说明

这个项目里反复出现的一个主题是：**自适应模板更新是一把双刃剑。**

- 当 `template_alpha = 0`（不更新）时，追踪器始终与最初那一帧（人工标注质量最高）进行比较。这种方式对短时遮挡和背景干扰比较鲁棒，但无法适应真实发生的外观变化（光照、姿态、尺度变化）。
- 当 `template_alpha > 0` 时，模板会逐帧融合更新（`template = (1-α)·template + α·new_patch`）。这种做法在正常工作时看起来更"丝滑"，但如果某一帧发生误匹配（遮挡、背景边缘干扰等），这个错误就会被固化进模板——而且由于这是一个正反馈过程，追踪器可能因此永久性地漂移，且没有自我修正的机制。

一些值得尝试的缓解方向（部分已经在讨论/原型阶段，尚无定论哪个是"最优解"）：
- 只有当匹配分数明显高于阈值时才更新模板，避免在勉强及格的匹配上也进行更新；
- 同时维护一个冻结的初始模板和一个自适应模板，将两者的打分结果结合使用；
- 维护一个最近被接受的候选块的滑动窗口，只有当新的更新与最近历史的*中位数*表现一致时才接受，而不是盲目信任任何单一的"标准答案"模板。

如果你尝试了上述某个方向并在具体序列上取得了不错的效果，欢迎提交 PR 分享你的发现（最好附上前后对比的 IoU 数据）。

## 评测指标

- **IoU**（交并比）：预测框与 ground-truth 框的交并比。
- **CLE**（中心位置误差）：预测框与 ground-truth 框中心点之间的欧氏距离（像素单位）。
- **成功率**：IoU 超过某个可配置阈值（默认 0.5）的帧所占的比例。

## 已知局限

- 没有尺度自适应——预测框的尺寸固定为模板尺寸（或轮廓的外接矩形），因此当目标在画面中明显变大或变小时，追踪器表现会下降。
- 长时间遮挡后的重新检测能力有限，目前只有简单的"连续丢失 N 帧后扩大搜索半径"逻辑。
- 相比现代的学习型特征，颜色/边缘特征的表达能力较弱，因此效果高度依赖具体序列（在部分 OTB100 序列上表现不错，在背景杂乱或遮挡严重的序列上表现较差）。
- 尚未针对完整的 OTB100 成功率/精度曲线（success/precision plots）进行基准测试，目前只打印简单的逐序列汇总统计。
